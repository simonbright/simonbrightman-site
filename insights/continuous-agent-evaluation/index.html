The Ghost in the Production Pipeline

Your agent passed every eval in staging. It is still failing in production — quietly, expensively, and at scale. The problem is not the model. It is the certification theater we built around it.

Simon Brightman · June 18, 2026 · 12 min read

There is a failure mode hiding inside every production agent deployment. I call it the ghost: behavior that never appears in your offline eval suite, never triggers your observability alerts, and never shows up in a single-turn benchmark — but manifests the moment a real user applies pressure across multiple turns, ambiguous intent, and operational edge cases.

The ghost is not a hallucination in the model weights. It is a structural gap in how we certify agents for production. We borrowed the evaluation playbook from static ML — holdout sets, accuracy scores, pass/fail gates — and applied it to systems that are fundamentally procedural, stateful, and non-deterministic. The result is agents that earn green dashboards on Friday and generate refund loops by Monday.

The certification theater problem

Most enterprise teams I work with have an eval story that sounds responsible on paper. They have golden datasets. They run regression tests before deploy. They may even use an LLM-as-judge to score response quality. Leadership sees a number — 87% pass rate — and approves the release.

But that number measures a frozen moment. A single prompt, a single expected output, a single scoring rubric applied once. Production agents do not operate in frozen moments. They operate in conversations where context accumulates, user frustration escalates, tool permissions matter, and prior errors compound downstream decisions.

A model can be wrong silently. An agent can be wrong loudly — wrong refund, wrong email, fifty dollars burned in tokens chasing its own tail.

The ghost lives in the delta between those two failure geometries. And because your eval infrastructure was never designed to stress-test multi-turn trajectories, the ghost ships with every release that cleared a single-turn gate.

Where agents actually die

Static code fails at syntax boundaries. Agents fail at behavioral boundaries — the seams where policy, tone, tool selection, and recovery logic intersect. In production, these are predictable categories:

  • Escalation mishandling — the agent fails to recognize when a human handoff is required, or escalates too late after trust is already lost.
  • Boundary drift — off-topic prompts gradually pull the agent outside its authorized scope without a clean refusal.
  • Tool-use miscalibration — calling the wrong tool, calling the right tool with wrong parameters, or looping tool calls without termination logic.
  • Error recovery collapse — a recoverable failure in turn two poisons turns three through ten because the agent cannot reset conversational state.
  • Compliance gaps — missing disclosures, incorrect refund policy application, or channel-inappropriate tone under adversarial user pressure.

None of these failure modes reliably surface in a single-turn eval row. They require adversarial, multi-turn scenario libraries that map directly to operational failure vectors — the same way pilots do not certify on one takeoff; they do check-rides.

Observability is not evaluation

A common misdirection: conflating distributed trace logging with behavioral certification. Tools like LangSmith and Datadog tell you what happened after the fact — latency spikes, token burn, tool invocation counts. They are essential for operations. They are not a substitute for knowing whether the agent should have behaved that way.

Observability is post-mortem infrastructure. Evaluation is pre-deployment and continuous certification infrastructure. When teams skip the second layer, they discover agent failures through customer complaints, chargebacks, and executive escalations — the most expensive eval signal in the stack.

What continuous evaluation actually means

Continuous agent evaluation is not running the same 200 prompts every night. It is maintaining a living library of multi-turn adversarial scenarios — escalation arcs, hostile customer recovery paths, tool-use stress cases — and running multi-model evaluation sweeps against deterministic heuristic rubrics that produce auditable performance bands.

This is a deliberate rejection of the expensive pattern where one LLM judges another LLM. Deterministic rubrics — keyword thresholds, structural heuristics, compliance phrase detection — are cheaper, repeatable, and debuggable. When a run fails, you get phrase highlights pointing to the exact turn where behavior diverged, not a opaque judge score with no remediation path.

Side-by-side diffing before you deploy

The highest-leverage moment in agent development is the prompt change or model upgrade that almost ships because offline spot-checks looked fine. Production teams need parallel run linking — the ability to execute the same scenario library against baseline and candidate configurations and diff operational compliance scores before merge.

This is the difference between "we think GPT-4.1-mini is cheaper and fine" and "GPT-4.1-mini fails escalation compliance on 14% of our support scenarios, concentrated in refund-exception flows." Without side-by-side diffing, you are A/B testing in production with real customers as the judge.

Scorecards stakeholders can actually read

Engineers live in traces and JSON. Executives live in risk exposure and brand liability. A functional eval platform must translate multi-turn behavioral results into human-readable scorecards — pass bands by scenario category, trend lines across model versions, and explicit compliance flags — without forcing non-engineering stakeholders to parse raw transcripts.

This is how agent governance matures from "the ML team says it is fine" to "Legal signed off on sales-agent disclosure compliance for this release." The ghost disappears when behavioral certification becomes a first-class gate in the release process, not a post-launch forensic exercise.

Build the harness before the agent

The teams that avoid the ghost share one discipline: they build evaluation infrastructure before they scale agent surface area. They treat scenario authorship as product work — domain experts, compliance reviewers, and engineers co-author failure vectors for customer support, sales qualification, internal copilots, and regulated vertical agents.

They name failure modes explicitly. They layer metrics: outcome correctness, trajectory quality, and operational cost. They roll out shadow → canary → full deployment. And they accept that agent eval is never done — it is continuous check-rides for systems whose behavior drifts with every model update, prompt tweak, and tool addition.

This is the architectural pattern behind work I have been doing through Vantage RuntimeAI — evaluation infrastructure purpose-built for production agent pipelines, not another observability dashboard or API gateway masquerading as quality assurance.

The takeaway

If your agent eval is a spreadsheet of single-turn prompts and a green CI badge, you are not measuring production readiness. You are performing certification theater while the ghost waits in the pipeline.

The fix is not a better model. It is a better eval substrate: adversarial multi-turn scenarios, deterministic rubrics, side-by-side model diffing, and scorecards that non-engineers can act on. Build that harness before the next release — or accept that your users will evaluate the agent for you, one expensive incident at a time.
